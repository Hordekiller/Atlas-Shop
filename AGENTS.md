## دستورالعمل: مهاجرت از SQLite به PostgreSQL

### مرحله ۱: نصب PostgreSQL و آماده‌سازی

- PostgreSQL را به‌صورت local نصب کن (یا Docker بالا بیاور).
- یک دیتابیس خالی برای پروژه بساز (مثلاً atlas_shop).

### مرحله ۲: اصلاح schema.prisma

- در بلاک datasource db:
  - provider را از "sqlite" به "postgresql" تغییر بده.
- کل schema را بازبینی کن و موارد ناسازگار با SQLite را اصلاح کن:
  - فیلدهایی که در SQLite به‌اجبار String بودند ولی باید enum باشند، به enum واقعی Prisma تبدیل کن.
  - فیلدهای آرایه‌ای که در SQLite به‌صورت String (مثلاً JSON یا comma-separated) ذخیره می‌شدند، در صورت امکان به نوع بومی Type[] یا Json تبدیل کن.
  - فیلدهای تاریخ را با @db.Timestamptz تنظیم کن (مهم برای منطقهٔ زمانی صحیح).
  - فیلدهای عددی پولی را با نوع Decimal @db.Decimal(12, 0) (یا دقت مناسب) تعریف کن، نه Float (مهم برای دقت مالی).

### مرحله ۳: اصلاح فایل محیطی

- در .env مقدار DATABASE_URL را به فرمت PostgreSQL تغییر بده:
  DATABASE_URL="postgresql://USER:PASSWORD@localhost:5432/atlas_shop?schema=public"

- یک فایل `.env.example` بدون مقادیر واقعی هم بساز/به‌روزرسانی کن.

### مرحله ۴: بازسازی کامل migration‌ها (چون داده تستی است)

- پوشهٔ قدیمی `prisma/migrations` را کاملاً حذف کن (migrationهای sqlite قابل استفاده روی postgres نیستند).
- migration اولیه را از صفر بساز:

  npx prisma migrate dev --name init

- سپس Prisma Client را تولید کن:

  npx prisma generate

### مرحله ۵: بازبینی کد برنامه (مهم)

بعد از تغییر نوع‌ها، کد را برای موارد زیر بررسی کن:

- هر جایی که آرایه/enum را قبلاً به‌صورت دستی **parse/stringify** می‌کردی (مثل `JSON.parse(field)`)، حالا باید حذف شود چون Prisma خودش نوع بومی برمی‌گرداند.
- query‌هایی که به رفتار خاص SQLite وابسته بودند (مثل عدم حساسیت پیش‌فرض به حروف یا توابع خاص).
- مقایسه‌های تاریخی و فیلترهای محدوده‌ای.

### مرحله ۶: Seed

- اسکریپت seed را اجرا کن تا داده‌های تستی روی PostgreSQL ساخته شوند:

  npx prisma db seed
  `

- مطمئن شو seed با نوع‌های جدید (enum/array/decimal) سازگار است.

### مرحله ۷: تست نهایی

- سرور را بالا بیاور و مسیرهای اصلی را تست کن: لیست محصولات، سبد خرید، ثبت سفارش، پرداخت.
- مخصوصاً منطق پرداخت و مبالغ را با نوع Decimal جدید تست کن.

## پرامپت: آماده‌سازی کامل پروژه برای انتشار روی GitHub

تو به کل ریپازیتوری‌های گیت‌هاب من دسترسی داری. وظیفهٔ تو این است که پروژهٔ «فروشگاه اطلس» (استک: NestJS, Next.js, Prisma, Turborepo) را به‌صورت کاملاً حرفه‌ای و استاندارد برای انتشار عمومی روی GitHub آماده کنی. تمام مراحل زیر را دقیق و کامل اجرا کن:

### ۱. پاک‌سازی پروژه از فایل‌های اضافی

- تمام فایل‌ها و پوشه‌های غیرضروری را شناسایی و حذف کن:
  - node_modules, dist, build, .next, .turbo, coverage
  - فایل‌های لاگ (_.log, npm-debug.log_, yarn-error.log)
  - فایل‌های موقت و سیستمی (.DS_Store, Thumbs.db, \*.tmp)
  - فایل‌های IDE شخصی (.idea, .vscode به‌جز تنظیمات تیمی مفید)
- هشدار امنیتی: مطمئن شو هیچ فایل .env واقعی، کلید، توکن یا پسوردی commit نشده باشد. اگر در تاریخچهٔ git چنین چیزی هست، به من هشدار بده.

### ۲. ساخت/به‌روزرسانی .gitignore

یک .gitignore جامع و مناسب monorepo (Turborepo) بساز که شامل موارد زیر باشد:

# Dependencies

node_modules/

# Builds

dist/
build/
.next/
.turbo/
out/

# Env

.env
.env.local
.env.\*.local

# Logs

_.log
npm-debug.log_

# OS / IDE

.DS_Store
Thumbs.db
.idea/

# Prisma

_.db
_.db-journal

# Coverage

coverage/

### ۳. فایل .env.example

- یک .env.example کامل بساز که تمام متغیرهای محیطی مورد نیاز پروژه را با مقادیر placeholder (نه واقعی) لیست کند:
  - DATABASE_URL, JWT_SECRET, کلیدهای درگاه پرداخت، SMS/OTP، و... .
- بالای هر بخش کامنت توضیحی بگذار.

### ۴. فایل README.md کامل و حرفه‌ای

یک README جامع به دو زبان فارسی و انگلیسی (یا حداقل فارسی کامل) بنویس که شامل این بخش‌ها باشد:

- عنوان و لوگو/بنر پروژه + توضیح کوتاه.
- Badges (لایسنس، نسخه، تکنولوژی‌ها).
- معرفی پروژه: فروشگاه اطلس چیست و چه می‌کند.
- ویژگی‌ها (Features): لیست کامل امکانات (صفحه‌ساز، پنل ادمین، پرداخت، و...).
- استک تکنولوژی: NestJS, Next.js, Prisma, PostgreSQL, Turborepo.
- پیش‌نیازها: Node نسخهٔ موردنیاز، PostgreSQL، pnpm/npm.
- راه‌اندازی (Installation): گام‌به‌گام و قابل کپی:
  - clone، نصب وابستگی‌ها، تنظیم .env، اجرای migration، seed، اجرای پروژه.
- ساختار پروژه (Project Structure): درخت پوشه‌ها با توضیح کوتاه.
- اسکریپت‌های مهم: دستورات dev/build/lint/test.
- اسکرین‌شات‌ها: بخش مخصوص برای تصاویر (placeholder بگذار).
- نقشهٔ راه (Roadmap): کارهای انجام‌شده و آینده.
- مشارکت (Contributing): نحوهٔ مشارکت.
- لایسنس.

### ۵. لایسنس (LICENSE)

- یک فایل LICENSE با لایسنس MIT بساز (مگر اینکه بگویم لایسنس دیگری می‌خواهم).
- سال و نام مالک (نام من / فروشگاه اطلس) را درست وارد کن.

### ۶. فایل‌های استاندارد اضافی

- CONTRIBUTING.md: راهنمای مشارکت.
- CODE_OF_CONDUCT.md: آیین‌نامهٔ رفتاری.
- SECURITY.md: نحوهٔ گزارش آسیب‌پذیری امنیتی.
- .github/ISSUE_TEMPLATE/: قالب برای bug report و feature request.
- .github/PULL_REQUEST_TEMPLATE.md: قالب Pull Request.
- در صورت نیاز CHANGELOG.md اولیه.

### ۷. بررسی نهایی کیفیت

- مطمئن شو package.json در root و هر app/package، فیلدهای name, version, description, license, author, repository را درست دارد.
- lint و format پروژه را بزن تا کد تمیز باشد.
- مطمئن شو پروژه بدون خطا build می‌شود.

### ۸. Commit و Push

- تمام تغییرات را با commit message‌های معنادار و استاندارد (Conventional Commits) ثبت کن:
  - مثال: chore: add gitignore and project cleanup, docs: add comprehensive README and license
- اگر ریپازیتوری روی گیت‌هاب وجود ندارد، یک ریپازیتوری جدید بساز (نام: atlas-shop یا هرچه مناسب‌تر است).
- فقط فایل‌های سالم و ضروری را push کن. هیچ فایل حساس یا اضافی نباید برود.
- در پایان لینک ریپازیتوری و خلاصه‌ای از کارهای انجام‌شده را به من بده.

### قوانین حیاتی

- هیچ secret/کلید/پسوردی نباید commit یا push شود.
- قبل از حذف هر فایل مشکوک، اگر مطمئن نیستی، از من بپرس.
- اگر در history گیت اطلاعات حساسی پیدا کردی، push نکن و اول به من اطلاع بده.
